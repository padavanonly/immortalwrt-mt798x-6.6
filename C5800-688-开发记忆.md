## 三、源码树内已做的关键修改

### 1. mt_wifi 改为使用原厂预编译模块

修改文件：`package/mtk/drivers/mt_wifi/Makefile`

处理思路：

- 不再依赖当前源码树重新编译 `mt_wifi.ko`
- 直接在 `Build/Compile` 阶段把原厂预编译 `.ko` 复制到 OpenWrt 期望的标准打包路径
- 保持 `KernelPackage` 的 `FILES` 指向标准打包目录，避免 `.ipk` 生成失败

当前 `kmod-mt_wifi` 打进去的模块路径逻辑为：

- `$(PKG_BUILD_DIR)/mt_wifi_ap/mt_wifi.ko`
- `$(PKG_BUILD_DIR)/mt_wifi/embedded/plug_in/warp_proxy/mtk_warp_proxy.ko`

说明：

- `conninfra.ko` 继续由 `kmod-conninfra` 提供
- `mtk_warp.ko` 继续由 `kmod-warp` 提供
- `kmod-mt_wifi` 只负责 `mt_wifi.ko` 和 `mtk_warp_proxy.ko`

### 2. 修复 688 DTS include 路径

修改文件：`target/linux/mediatek/dts/mt7981-nradio-common.dtsi`

修复内容：

- 原来是 `#include "../mt7981.dtsi"`
- 已改为 `#include "mt7981.dtsi"`

原因：

- 镜像生成阶段编译 `mt7981-nradio-c5800-688.dtb` 时，报错找不到 `../mt7981.dtsi`
- 修正后，`target/linux/install` 已能正常通过

### 5. 已定位的开机失败原因

这次 `C5800-688` 不能开机，已经不是单纯 `mtwifi` 缺文件，而是低层驱动和最终镜像打包链路没有完全对齐旧可开机固件。

已明确的原因如下：

1. 最早的失败并非 `DTS`、`bootargs` 或 `CONTROL` 明显错误。
   - `DTS` 关键项、`root=PARTLABEL=rootfs_2nd`、`BOARD=nradio_c5800-688` 后来都已经对齐。

2. 真正的关键差异在低层模块。
   - 旧可开机固件与新固件最初不一致的模块有：
     - `conninfra.ko`
     - `mtk_warp.ko`
     - `mt_wifi.ko`
   - 同时新固件比旧可开机固件多自动加载了：
     - `mtk_warp_proxy.ko`

3. 之前虽然已经把正确模块放进 `staging_dir/.../root-mediatek`，但最终镜像实际打包用的是：
   - `build_dir/target-aarch64_cortex-a53_musl/root-mediatek`
   - 也就是说，前面一段时间“以为已经对齐”的模块，其实没有真正进最终 `sysupgrade.bin`。

4. 在把 `build_dir/.../root-mediatek` 里的以下文件也强制对齐到旧可开机固件后，低层模块层面才算真正对齐：
   - `lib/modules/6.6.95/conninfra.ko`
   - `lib/modules/6.6.95/mtk_warp.ko`
   - `lib/modules/6.6.95/mt_wifi.ko`
   - `etc/modules.d/mt_wifi`

当前结论：

- 开机失败的主因是“最终镜像里的低层无线/warp/conninfra 模块没有真正对齐旧可开机固件”。
- 不是单纯 LuCI、插件或普通软件包多少的问题。

### 10. 已定位的 modem 页面/模组异常原因

`modem` 页面和模组状态异常的根因已经明确，不是插件脚本本身：

1. 最早 USB/XHCI 底层初始化不完整
   - `C5800-688` 的 DTS 最初只开启了 `xhci`，但缺少 `usb_phy` 和 `phys` 相关配置
   - 导致 USB 主控虽然部分起来，但模组链路不稳定

2. 系统缺少 USB 运行时依赖
   - 后续已补入：
     - `adb`
     - `usbutils`
     - `usbids`
     - `libusb-1.0`
     - `libudev-zero`
     - `libevdev`

3. 当时的直接现象是：
   - `lsusb` 报 `unable to initialize libusb: -99`
   - `/tmp/modconf.conf` 显示 `未检测到模组`
   - `modeminit` 日志持续出现 `No response from modem`

4. 修复后已确认：
   - `lsusb` 正常
   - 模组型号识别正常
   - `modem` 页面恢复正常

结论：

- 此问题属于底层 USB/XHCI/DTS 和系统依赖链问题，不属于 5G 插件脚本逻辑问题
- 后续禁止再对 5G 插件源码进行“误判式修改”
